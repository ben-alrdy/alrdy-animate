title: aa-trigger
description: Override the default scroll trigger on any animated element. Supported values are `scroll`, `click`, `event:<name>`, `load`, `load-once`, and `lcp` (load-timed, optimised for Largest Contentful Paint). Combine values with a space (e.g. `load-once event:enter`) to play on first page load AND on subsequent page-transition events.

`aa-trigger` controls *when* an animation fires. By default every animation is `scroll`-driven (ScrollTrigger), but for hero content above the fold ‚Äî where the element is already in view at page load ‚Äî you usually want it to play immediately.

## Values

| Value           | Behaviour                                                                                              |
| --------------- | ------------------------------------------------------------------------------------------------------ |
| `scroll`        | Default. Plays when the element scrolls into view (or scrubbed if `aa-scrub` is set).                  |
| `load`          | Plays on **every** `init()` call, including the first. Use for SPA route entries (Next.js App Router back/forward nav). Honours `aa-delay` for sequencing siblings. |
| `load-once`     | Plays on the **first** `init()` call in the page session ‚Äî never on subsequent inits. Use for hero content under a page-transition wrapper (Barba, View Transitions). Honours `aa-delay` like `load`. |
| `lcp`           | Like `load`, but for the **Largest Contentful Paint** element (the hero's largest image/visual). Painted at first paint at `0.01` opacity so it counts toward LCP *before* the bundle loads, then fades to full. See [Optimizing LCP](#optimizing-lcp-with-aa-triggerlcp) below. |
| `instant`  | A **CSS-driven** entrance that paints on the first frame, *before* GSAP loads ‚Äî the hero feels instant. The library builds no GSAP tween (it's owned by an inline keyframe). Element-level motion from `aa-animate`; add the loader for `aa-delay` staircases and per-character `text-*` cascades. See [Instant hero entrances](/recipes/instant-hero/). |
| `event:<name>`  | Stays paused until an `aa:trigger` CustomEvent with `detail.name === <name>` is dispatched on it.      |
| `click`         | Plays on click of the element.                                                                         |

Inside known containers (`[aa-modal-name]`, `[aa-tabs-content]`, `[aa-tabs-visual]`, `[aa-slider-item]`) the trigger is *inferred* ‚Äî e.g. animations inside a slide automatically use `event:slide-active`. Setting `aa-trigger` explicitly always wins.

`load` and `load-once` entrances are **paint-gated**: the lib builds them paused and starts them one frame after the reveal, so the animation begins exactly at its first painted frame instead of jumping ahead while a heavy page is still laying out. The global [`loadDelay`](/initialization/init-options/) init option (default `0`) adds an optional beat on top of any per-element `aa-delay`, measured from that paint ‚Äî set a positive value if you want the entrance to hold briefly before playing.

## Why `load-once` exists

If `load-once` fired on every `init()` call, every Barba (or View Transition) navigation would replay the animation **while the new container is still hidden behind the transition wrapper** ‚Äî the user never sees it, and by the time the wrapper drops away the element is already in its final state. So `load-once` is intentionally one-shot: it fires on the *very first* init in the page lifetime (until the next hard reload), then stays out of the way.

On subsequent inits, a `load-once`-only element gets **no animation** ‚Äî no scroll-trigger fallthrough, no from-state applied. The `aa-ready` flip at the end of `init()` makes it visible in its natural CSS state. This is exactly what you want for hero text under a page-transition wrapper: animate beautifully on first load, then just be present on every navigation back.

Combine with another trigger to handle subsequent navigations explicitly ‚Äî see below.

## Combining triggers

Space-separated values run as a list. The `load-once` trigger always owns the first init cycle (and short-circuits any other triggers on that element so they can't double-fire). Non-`load-once` triggers wire up normally on **subsequent** inits. This is the canonical pattern for hero content in a page-transition setup:

```html
<h1
  aa-animate="text-slide-up"
  aa-split="lines mask"
  aa-trigger="load-once event:enter"
>
  Hero headline
</h1>
```

On first page load: `load-once` fires immediately (or the [CSS fallback](/recipes/load-fallback/) plays if the bundle is slow). On every subsequent Barba navigation: the recipe's leave timeline dispatches `aa:trigger { name: 'enter' }` at the visual peak, and the headline animates then. No double-fire, no replay-while-hidden.

See the [Webflow + Barba recipe](/recipes/webflow-barba/) for the full pattern.

## Slow-network fallback

`aa-trigger="load-once"` only fires once `init()` runs, which means once the JS bundle has downloaded, parsed, and called the lib. On a slow connection that delay is user-visible ‚Äî and the lib's FOUC guard (`visibility: hidden` on `[aa-animate]`) keeps the hero invisible the whole time.

For Webflow / `<script>`-tag projects loading from a CDN, drop the [load-fallback recipe](/recipes/load-fallback/) into `<head>`. After a tunable timeout, hero elements fade up via CSS ‚Äî preserving the same staircase you defined with `aa-delay` and `aa-stagger`. When the bundle eventually arrives the lib detects the fallback and skips its own animation, so nothing flashes or replays. (Requires `defer` on your CDN scripts so the body parses before the timer fires ‚Äî see the recipe.)

## Optimizing LCP with `aa-trigger="lcp"`

A fade-in on your hero's largest image **hurts your Largest Contentful Paint score**. Chrome excludes elements that are `opacity: 0` or `visibility: hidden` from LCP measurement ‚Äî and a normal `aa-trigger="load"` entrance is *both*: the FOUC guard hides it (`visibility: hidden`) and the fade starts at `opacity: 0`. So LCP can't be recorded until the JS bundle has downloaded, parsed, run `init()`, and started the fade. On a slow connection that's a poor LCP, even though the image was ready much earlier.

`aa-trigger="lcp"` fixes this. The companion stylesheet paints the marked element at first paint at a near-invisible `opacity: 0.01` ‚Äî non-zero, so Chrome counts it as an LCP candidate **immediately, before the bundle runs** ‚Äî and skips the FOUC guard for it. When the lib loads, the entrance fades it from `0.01` to full. Visually identical to a fade from `0`; to Lighthouse, the LCP is recorded at first paint.

```html
<img
  aa-animate="fade"
  aa-trigger="lcp"
  fetchpriority="high"
  src="hero.jpg"
  alt="‚Ä¶"
/>
```

It works with any opacity-bearing `aa-animate` ‚Äî `fade`, `rotate-up`, `blur`, `fade-up`, etc. ‚Äî so the LCP element can still have a rich entrance, not just a plain fade.

<Demo>
  <div class="lcp-demo">
    <div class="lcp-demo__box lcp-demo__fade" aa-animate="fade" aa-trigger="lcp">fade ¬∑ lcp</div>
    <div class="lcp-demo__box lcp-demo__rotate" aa-animate="rotate-up" aa-trigger="lcp">rotate-up ¬∑ lcp</div>
  </div>
</Demo>

**Use it correctly:**

- Put it on a **single** above-the-fold element ‚Äî the one that is your LCP (usually the largest hero image or visual). Marking several elements defeats the purpose.
- Keep the image in the **static HTML** (not injected by JS) and add `fetchpriority="high"` (or `loading="eager"`) so the browser fetches it early ‚Äî `aa-trigger="lcp"` only removes the library's own LCP regression; it can't speed up an image the browser doesn't know about yet.
- **Requires the updated inline `<head>` snippet.** The LCP rules must apply at first paint, so they live in the inline snippet you paste, not the non-blocking external stylesheet. If you set up this site before `lcp` existed, re-paste the [head snippet](/installation/webflow/#site-wide-head-code) (or the [load-fallback recipe](/recipes/load-fallback/)).

Under the slow-network fallback the element still reveals: if `init()` is slow, it fades in via CSS (`aa-fallback`); if the bundle never arrives, it's shown at full opacity (`aa-timeout`).

## Instant hero entrances with `aa-trigger="instant"`

Where `lcp` optimises the *one* largest image, `instant` makes a whole hero's **entrance** paint on the first frame as pure CSS ‚Äî before GSAP loads ‚Äî so there's no wait for the bundle at all. The motion comes from `aa-animate` (element-level `fade` / `fade-up` / `blur` / `zoom` / `slide` / `rotate`); when the bundle lands the library skips the element entirely (no GSAP tween, no flash). An optional ~0.9 KB loader script adds an `aa-delay` staircase and instant per-character `text-*` cascades. Line-based text effects (slide-up/tilt/oval) stay on the GSAP path.

It needs an inline `<head>` block (the keyframe + reveal rules must be present at first paint) ‚Äî the full copy-paste setup is in the **[Instant hero entrances recipe](/recipes/instant-hero/)**.

## Page transitions (Barba, View Transitions)

`aa-trigger="load-once"` fires *only* on the very first `init()` call in the page session ‚Äî on subsequent inits (Barba navigations, programmatic `destroy()` + `init()`) it's a no-op and the element renders in its natural state. That's almost always what you want for hero text under a transition wrapper: animate beautifully on first load, then just be present on every navigation back.

If you need an animation on every navigation too, combine `load-once` with `event:enter` (`aa-trigger="load-once event:enter"`) ‚Äî `load-once` handles first init, `event:enter` handles every subsequent nav. See the [Webflow + Barba recipe](/recipes/webflow-barba/) for the full pattern.

The slow-network fallback only matters for the first page load ‚Äî subsequent navigations have the bundle cached, so init runs immediately.

## SPA route entry (Next.js App Router) ‚Äî use `load`

In a Next.js App Router setup, a single `AlrdyInit` client component lives in the root layout and re-calls `init()` on every `usePathname()` change. There's no transition wrapper hiding the leaving DOM ‚Äî the new route is rendered cold, and the user expects the entrance animation **every time** they arrive on a "new" page (including via browser back/forward nav).

`aa-trigger="load-once"` is wrong here because it skips every init after the first; the headline lands at its final state with no animation. `aa-trigger="load"` is the right primitive:

```html
<h1 aa-animate="text-tilt-up" aa-trigger="load">Welcome to /cohort</h1>
<a aa-animate="fade-up" aa-trigger="load" aa-delay="0.2">Get started</a>
```

The animation fires on cold load, forward nav, and back/forward ‚Äî wherever `init()` runs.

| Scenario | Use |
|---|---|
| Webflow + Barba, View Transitions, any transition wrapper covers the swap | `load-once` (or `load-once event:enter` for both) |
| Next.js App Router, plain SPA, no transition wrapper | `load` |
| Mixed: first paint *and* an in-page event later (modal open, etc.) | combine, e.g. `load event:modal-active` |
