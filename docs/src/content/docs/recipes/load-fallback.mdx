---
title: Slow-network + init-timeout fallback
description: Optional inline <head> snippet with two safety layers ‚Äî a graceful hero fade if the JS bundle is late, and a universal reveal if init never arrives at all. First-load only ‚Äî Barba / View Transitions navigations are not at risk because the bundle is already cached.
---

The lib hides every `[aa-animate]` element with `visibility: hidden` until `init()` runs. On a slow connection (or if the bundle is blocked, errors, or is misconfigured), that leaves animated content invisible. This snippet adds two layers of safety:

| Layer | Default | What it does |
| --- | --- | --- |
| **`aa-fallback`** | 1s | Fades `aa-trigger="load"` / `aa-trigger="load-once"` elements (hero copy) via a CSS keyframe ‚Äî graceful staircase, respecting `aa-delay` and `aa-stagger`. |
| **`aa-timeout`** | 4s | Reveals **every** `[aa-animate]` element at its natural state with no animation ‚Äî universal safety net if init never runs. |

Drop this snippet into `<head>` *above* the alrdy-animate `<script>` and `<link>` tags ‚Äî and ideally above any other render-blocking `<script>` (e.g. Typekit), so the two timers start counting from page load rather than after that script downloads:

```html
<!-- alrdy-animate FOUC guard + slow-network/timeout safety (optional but recommended). -->
<script>
  const FALLBACK_DELAY = 1000;  // ms ‚Äî load elements fade in via CSS keyframe
  const TIMEOUT_DELAY  = 4000;  // ms ‚Äî universal reveal if init never runs

  const selectors = '[aa-trigger~="load"][aa-animate], [aa-trigger~="load-once"][aa-animate]';
  const styled = new WeakSet();

  const applyFallback = () => {
    if (document.documentElement.hasAttribute('aa-loaded')) return;
    document.documentElement.setAttribute('aa-fallback', '');
    document.querySelectorAll(selectors).forEach((el) => {
      if (styled.has(el)) return;  // apply once, even across polls
      styled.add(el);
      const baseDelay = parseFloat(el.getAttribute('aa-delay')) || 0;
      if (baseDelay > 0) el.style.animationDelay = baseDelay + 's';
    });
  };

  // Poll, not DOMContentLoaded: render-blocking scripts can delay that past these timers.
  const deadline = performance.now() + FALLBACK_DELAY + 8000;
  const tryFallback = () => {
    if (document.documentElement.hasAttribute('aa-loaded')) return;
    applyFallback();
    if (document.readyState === 'loading' && performance.now() < deadline) {
      setTimeout(tryFallback, 100);
    }
  };
  setTimeout(tryFallback, FALLBACK_DELAY);

  setTimeout(() => {
    if (document.documentElement.hasAttribute('aa-loaded')) return;
    document.documentElement.setAttribute('aa-timeout', '');
  }, TIMEOUT_DELAY);
</script>
<style>
  [aa-animate]:not([aa-ready]):not([aa-trigger~="lcp"]) { visibility: hidden; }
  @keyframes aa-fallback-appear {
    from { opacity: 0; transform: translateY(1.5rem); }
    to   { opacity: 1; transform: translateY(0); }
  }
  html[aa-fallback] [aa-trigger~="load"][aa-animate]:not([aa-ready]),
  html[aa-fallback] [aa-trigger~="load-once"][aa-animate]:not([aa-ready]) {
    visibility: visible;
    animation: aa-fallback-appear 0.5s ease both;
  }
  html[aa-timeout] [aa-animate]:not([aa-ready]) { visibility: visible; }
  /* LCP: paint marked element at 0.01 pre-bundle (eligible LCP candidate), then fade to full. */
  [aa-animate][aa-trigger~="lcp"]:not([aa-ready]) { visibility: visible; opacity: 0.01; }
  html[aa-fallback] [aa-animate][aa-trigger~="lcp"]:not([aa-ready]) { opacity: 1; transition: opacity 0.5s ease; }
  html[aa-timeout] [aa-animate][aa-trigger~="lcp"]:not([aa-ready]) { opacity: 1; }
</style>
<noscript>
  <style>
    /* JS disabled: neither init() nor the fallback timer runs, so the guard
       would hide everything permanently. Force all animated elements visible. */
    [aa-animate] {
      visibility: visible !important;
      opacity: 1 !important;
      transform: none !important;
      filter: none !important;
      clip-path: none !important;
    }
  </style>
</noscript>
```

Tune the two constants at the top to taste. Keep `TIMEOUT_DELAY` well above `FALLBACK_DELAY` ‚Äî the first is "let JS win the race for the hero", the second is "give up, show the page".

**Why the inline `<style>`** ‚Äî the FOUC guard and the `lcp` opacity floor also ship in `dist/alrdy-animate.css` (that's what the npm `import 'alrdy-animate/style'` path relies on), but that file lands *after* the HTML renders ‚Äî always on a slow connection, and deliberately so when you load it non-blocking (the [Webflow install guide](/installation/webflow/#1-site-wide-head-code) uses `media="print"` to keep it off the critical render path). Without inlining, every animated element flashes briefly visible, then disappears when the guard arrives, then reappears when `init()` runs. Inlining the critical CSS in `<head>` closes that window ‚Äî the FOUC guard is active from the very first paint, which is also what makes loading the external stylesheet non-blocking safe.

The fallback keyframe and the `aa-fallback` / `aa-timeout` reveal rules, by contrast, ship **only** in this snippet ‚Äî *not* in `dist/alrdy-animate.css`. They're inert anywhere the snippet isn't loaded (nothing else sets those attributes on `<html>`), so the dist file would only ever carry a dead, conflicting copy. That means the snippet's keyframe is the single source of truth ‚Äî no cross-origin dist copy can win a duplicate-`@keyframes` collision against your customisation.

## How the staircase works

The script honours one timing attribute so the fallback looks like a quieter version of the real animation:

- **`aa-delay="<seconds>"`** ‚Äî delays the element's fade-up. E.g. a paragraph with `aa-delay="0.2"` appears 200ms after the headline, so the hero's staircase still reads. CSS alone can't apply this (typed `attr()` isn't widely supported yet), so the script sets it inline.

Everything else fades as a whole element via the CSS keyframe. The script deliberately **ignores `aa-stagger`**: a stagger targets units that don't exist yet at fallback time ‚Äî the split-text lines/chars of a `text-*` animation, or the children the real init walks ‚Äî and the bundle hasn't run to create them. Trying to reproduce it risks animating the wrong nodes (e.g. only an inline `<span>` inside a headline, leaving the rest of the text to just appear). A clean whole-element fade is the safe quieter version; on a normal-speed load the bundle wins the race and the real per-line / per-child stagger plays instead.

## What each layer does

`init()` sets `aa-loaded` on `<html>` as its first synchronous step. The two timers check for that attribute and bail out if it's already set.

### Layer 1 ‚Äî `aa-fallback` (1s default)

Sets `html[aa-fallback]`. The CSS rule matches both `[aa-trigger~="load"]` and `[aa-trigger~="load-once"]`, so it covers every load-triggered element (including combined forms like `aa-trigger="load-once event:enter"`). The keyframe fades those elements up, respecting `aa-delay` and `aa-stagger` so the staircase still plays.

When the bundle eventually arrives, the lib's load-trigger branches detect `aa-fallback` and skip their JS animation entirely ‚Äî the elements stay where the fallback put them, no replay or flash. `init()` clears `aa-fallback` at the end, so subsequent Barba navigations don't re-trigger the keyframe.

Scoped to `[aa-trigger~="load"]` / `[aa-trigger~="load-once"]` on purpose: scroll-triggered elements below the fold stay hidden until the lib actually arrives. Only elements the author has explicitly marked as "needs to be visible immediately" are eligible for the fallback fade.

### Layer 2 ‚Äî `aa-timeout` (4s default)

Sets `html[aa-timeout]`. The CSS rule reveals **every** `[aa-animate]` element at its natural rendered state ‚Äî no animation, no from-state. This is the deep-fail safety net: if JS errored, the bundle is blocked, or the network is so slow that even the hero staircase isn't enough, the page still ends up usable.

When `init()` eventually arrives, it detects `aa-timeout` and switches to bypass mode:

- Drops `scroll`, `text`, `reveal`, `parallax` from the feature load set ‚Äî no from-states applied, no entrance ScrollTriggers, no flash from rewinding the now-visible elements.
- Skips the reduced-motion fade pass.
- Still wires up interactive features (tabs, slider, marquee, modal, nav, hover, cursor, split) so components remain functional.
- Still mounts smooth-scroll, scroll-state, and scroll-target globals.
- Flips `aa-ready` on every `[aa-animate]` element so the DOM ends in a consistent state.
- Clears `aa-timeout`, so the next Barba navigation runs animations normally.

The tradeoff: appearance animations don't play on this load (the user already saw the content for 4+ seconds), and event-triggered scroll/text/reveal entries don't fire on this page. At that point, page consistency outweighs animation correctness.

## Customising the motion

The default keyframe is a small upward translate (`translateY(1.5rem) ‚Üí 0`) plus opacity. To use a different fallback motion (slide-down, pure fade, scale-in), **edit `@keyframes aa-fallback-appear` directly in the snippet** ‚Äî it's the only definition, so there's nothing to override:

```css
@keyframes aa-fallback-appear {
  from { opacity: 0; transform: scale(0.95); }
  to   { opacity: 1; transform: scale(1); }
}
```

The duration, easing, and selector stay the same ‚Äî just the keyframe content changes. (The library no longer ships its own copy of this keyframe in `dist/alrdy-animate.css`. Earlier versions did, and because that cross-origin copy loaded *after* the snippet it won the duplicate-name collision and silently reverted your edit ‚Äî so a separate override never took. If you're customising on an older release, rename the keyframe **and** raise the rule's specificity, or upgrade.)

## Why the snippet polls instead of waiting for `DOMContentLoaded`

The obvious way to write the apply step is "wait until the DOM is ready, then style the hero" ‚Äî gate on `document.readyState` and fall back to `DOMContentLoaded`. **Don't.** On a real page it loses the race every time, for a reason that isn't obvious:

A render-blocking `<script src>` (no `defer`/`async`) pauses HTML parsing until it downloads. While the parser is paused, parsing is not "complete", so `document.readyState` stays `'loading'` and `DOMContentLoaded` does not fire. On a Webflow page there are always render-blocking scripts you don't control ‚Äî Typekit in `<head>`, and the **jQuery + `webflow.js` + GSAP/ScrollTrigger** bundle Webflow appends at the *end* of `<body>`. On a genuinely slow connection those keep `readyState` at `'loading'` for **5‚Äì15 seconds**, and push `DOMContentLoaded` *past* `TIMEOUT_DELAY` ‚Äî so the safety-net reveal has already fired and the staircase never plays. A `readyState`/`DOMContentLoaded` gate simply cannot win.

The fix is to stop using "document finished parsing" as a proxy for "the hero exists." The hero is near the top of `<body>` and is parsed within the first second or two regardless of what's blocking further down the page. So the snippet **polls for the load-trigger elements** and applies the moment they appear ‚Äî typically seconds before `DOMContentLoaded` on a slow load.

### Still add `defer` to your own CDN scripts

`defer` doesn't fix the snippet (you can't defer Webflow's injected scripts), but it's still worth doing on the `<script src="‚Ä¶">` tags *you* control:

```html
<script defer src="https://cdn.jsdelivr.net/npm/gsap@3/dist/gsap.min.js"></script>
<script defer src="https://cdn.jsdelivr.net/npm/gsap@3/dist/ScrollTrigger.min.js"></script>
<script defer src="https://cdn.jsdelivr.net/npm/lenis@1/dist/lenis.min.js"></script>
<script defer src="https://cdn.jsdelivr.net/npm/alrdy-animate@{{version}}/dist/alrdy-animate.umd.js"></script>
```

Deferred scripts don't block parsing, so the hero parses sooner (the staircase can start earlier) and `init()` runs sooner once the bundle lands. Two related wins on Webflow specifically: load alrdy-animate's GSAP build *instead of* Webflow's native GSAP rather than both ‚Äî shipping GSAP twice is a large chunk of the render-blocking weight that delays the real init ‚Äî and if you can, bind your `AlrdyAnimate.init(...)` call to the bundle being ready rather than to `DOMContentLoaded`, which on a slow Webflow load is one of the last events to fire.

## When you don't need this

The fallback only matters when something can leave the FOUC guard's `visibility: hidden` in place ‚Äî i.e. whenever `init()` can be slow or fail to run. You can skip the snippet if:

- **You don't use the FOUC guard at all** ‚Äî you don't load `alrdy-animate/style` (or otherwise apply the `[aa-animate] { visibility: hidden }` rule), and you accept a brief reflow as `init()` applies from-states. With nothing hiding the elements, there's nothing to reveal.
- **The bundle is inlined into the initial HTML payload** (not a separate `<script src>` / chunk) *and* you're comfortable with elements staying invisible in the rare case `init()` throws.

"It's a Vite / Next.js / Webpack app, not a CDN script" is **not**, on its own, a reason to skip it ‚Äî see [Using it in a bundled app](#using-it-in-a-bundled-app) below. A separately-loaded bundle is still subject to download latency and chunk-load failures, both of which leave guarded elements hidden.

If you only want one of the two layers, delete the other layer's `setTimeout(...)` (for `aa-fallback`, the `tryFallback` block and its scheduling `setTimeout`) and its corresponding CSS rule.

## Using it in a bundled app

The moment you load `alrdy-animate/style` in a Next.js / Vite / Webpack app, the same two risks apply: the FOUC guard hides every `[aa-animate]` element until `init()` runs, and `init()` waits on the JS bundle ‚Äî which a slow connection delays and a failed chunk-load (or an error inside `init()` itself) can prevent outright, leaving the SSR'd hero invisible while the rest of the page renders. So you want the same two layers.

One rule that's easy to get wrong: **the timer must run independently of your app bundle.** Put it in a `useEffect` or a top-level module import and it can't fire until the bundle loads ‚Äî which is the very thing it's meant to beat. It has to be inline in the document `<head>`, ahead of (or `beforeInteractive` relative to) your app's JS.

Use the **same `<script>` + `<style>` + `<noscript>`** as the snippet at the top of this page ‚Äî only the placement and one trim change:

- The base **FOUC guard** and **`lcp` opacity floor** already come from `import 'alrdy-animate/style'` (a render-blocking `<link>`, present at first paint), so you don't re-inline those two rules ‚Äî only the fallback keyframe and the `html[aa-fallback]` / `html[aa-timeout]` reveal rules, which no longer ship in `dist`.
- The `html[aa-fallback]` lcp rule out-specificities the imported `0.01` floor regardless of `<link>` order, so it still wins.

### Next.js App Router ‚Äî `app/layout.tsx`

```tsx
import 'alrdy-animate/style' // FOUC guard + lcp floor (render-blocking, at first paint)

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <head>
        {/* Fallback keyframe + reveal rules ‚Äî NOT in dist, so inline them here. */}
        <style dangerouslySetInnerHTML={{ __html: `
          @keyframes aa-fallback-appear { from { opacity: 0; transform: translateY(1.5rem) } to { opacity: 1; transform: none } }
          html[aa-fallback] [aa-trigger~="load"][aa-animate]:not([aa-ready]),
          html[aa-fallback] [aa-trigger~="load-once"][aa-animate]:not([aa-ready]) { visibility: visible; animation: aa-fallback-appear .5s ease both }
          html[aa-timeout] [aa-animate]:not([aa-ready]) { visibility: visible }
          html[aa-fallback] [aa-animate][aa-trigger~="lcp"]:not([aa-ready]) { opacity: 1; transition: opacity .5s ease }
          html[aa-timeout] [aa-animate][aa-trigger~="lcp"]:not([aa-ready]) { opacity: 1 }
        ` }} />
        {/* The timer ‚Äî paste the <script> body from the snippet at the top of this page. */}
        <script dangerouslySetInnerHTML={{ __html: `/* FALLBACK_DELAY / TIMEOUT_DELAY timer from the snippet above */` }} />
        {/* JS-disabled reveal ‚Äî same <noscript> as the snippet above. */}
        <noscript><style dangerouslySetInnerHTML={{ __html: `[aa-animate]{visibility:visible!important;opacity:1!important;transform:none!important;filter:none!important;clip-path:none!important}` }} /></noscript>
      </head>
      <body>{children}</body>
    </html>
  )
}
```

A raw `<script dangerouslySetInnerHTML>` in the `<head>` is injected into the SSR'd HTML and runs before hydration, so it beats the bundle without needing `next/script` ‚Äî but `<Script strategy="beforeInteractive">` from `next/script` works too. The timer only ever sets `aa-*` attributes on `<html>` and an inline `animation-delay` on plain DOM nodes ‚Äî neither is a React-controlled prop, so there's no hydration mismatch.

### Next.js Pages Router ‚Äî `pages/_document.tsx`

Put the same three tags inside `<Head>` in `_document.tsx` (using `dangerouslySetInnerHTML` for the `<script>` and `<style>`). Keep the `import 'alrdy-animate/style'` in `_app.tsx`.

### Plain Vite / SPA

Paste the snippet verbatim into your `index.html` `<head>`, exactly like the Webflow block ‚Äî there's no async-import subtlety, the inline `<style>` is the single source.

## Page transitions (Barba)

This snippet only protects the *first* page load. On subsequent navigations the bundle is cached and `init()` runs in `beforeEnter` parallel to the leave timeline (see the [Webflow + Barba recipe](/recipes/webflow-barba/)). `aa-loaded` persists across navigations, so the fallback timer's check passes immediately and nothing fires.

## Verifying it locally

### Layer 1 ‚Äî `aa-fallback` (slow network)

1. Open DevTools ‚Üí Network ‚Üí check "Disable cache" ‚Üí throttle to "Slow 3G" (or "Slow 4G").
2. Reload. You should see:
   - Hero elements stay invisible for `FALLBACK_DELAY` plus however long the hero takes to parse ‚Äî roughly 1‚Äì3s on a slow connection, not a hard 1s.
   - They then fade up ‚Äî respecting any `aa-delay` and `aa-stagger` you set, so the staircase plays out before the bundle finishes downloading.
   - This should land *before* the `aa-timeout` reveal (4s default). If the staircase only ever appears as an instant, no-animation reveal at the 4s mark, the fallback isn't winning ‚Äî see the troubleshooting note below.
   - When the bundle eventually lands, no second animation plays ‚Äî the elements stay where the fallback put them.
3. Open the Elements panel and inspect a load-trigger element while the fallback is in flight. You should see `<html aa-fallback>`, and on any element with `aa-delay`, an inline `style="animation-delay: 0.2s"`. Elements without `aa-delay` fade via the CSS keyframe with no inline style. If `aa-fallback` is set but a delayed element's inline style is missing, the timer fired before that element parsed ‚Äî the poll retries and applies it once it appears.

If the staircase never plays and you only get the blanket 4s reveal, you almost certainly have an older copy of this snippet that gated the apply step on `document.readyState === 'loading'` / `DOMContentLoaded`. On any page with render-blocking scripts (every Webflow page) that gate defers the apply until `DOMContentLoaded`, which fires *after* the timeout ‚Äî re-copy the current snippet, which polls instead.

### Layer 2 ‚Äî `aa-timeout` (bundle never arrives)

1. In DevTools ‚Üí Network, **block** the alrdy-animate `<script>` URL (right-click the request ‚Üí Block request URL).
2. Reload. You should see:
   - All `[aa-animate]` elements stay hidden for ~4s (the `TIMEOUT_DELAY` interval).
   - At the timer mark, every animated element on the page becomes visible at its natural rendered state ‚Äî no fades, no entrance motion.
   - The Elements panel shows `<html aa-timeout>`.
3. Unblock the request and reload. Hero fades in via Layer 1 if it gets stuck under 4s, otherwise normal init runs and the snippet does nothing.
