---
title: Slow-network + init-timeout fallback
description: Optional inline <head> snippet with two safety layers — a graceful hero fade if the JS bundle is late, and a universal reveal if init never arrives at all. First-load only — Barba / View Transitions navigations are not at risk because the bundle is already cached.
---

The lib hides every `[aa-animate]` element with `visibility: hidden` until `init()` runs. On a slow connection (or if the bundle is blocked, errors, or is misconfigured), that leaves animated content invisible. This snippet adds two layers of safety:

| Layer | Default | What it does |
| --- | --- | --- |
| **`aa-fallback`** | 1s | Fades `aa-trigger="load"` / `aa-trigger="load-once"` elements (hero copy) via a CSS keyframe — graceful staircase, respecting `aa-delay` and `aa-stagger`. |
| **`aa-timeout`** | 4s | Reveals **every** `[aa-animate]` element at its natural state with no animation — universal safety net if init never runs. |

Drop this snippet into `<head>` *above* the alrdy-animate `<script>` and `<link>` tags — and ideally above any other render-blocking `<script>` (e.g. Typekit), so the two timers start counting from page load rather than after that script downloads:

```html
<!-- Slow-network + init-timeout safety (optional but recommended).
     aa-fallback fades aa-trigger="load"/"load-once" elements at FALLBACK_DELAY
     (graceful hero staircase). aa-timeout reveals every animated element at
     TIMEOUT_DELAY (universal safety net if init never arrives). -->
<script>
  const FALLBACK_DELAY = 1000;  // ms — load elements fade in via CSS keyframe
  const TIMEOUT_DELAY  = 4000;  // ms — universal reveal if init never runs

  const selectors = '[aa-trigger~="load"][aa-animate], [aa-trigger~="load-once"][aa-animate]';
  const styled = new WeakSet();

  const applyFallback = () => {
    if (document.documentElement.hasAttribute('aa-loaded')) return;
    document.documentElement.setAttribute('aa-fallback', '');
    document.querySelectorAll(selectors).forEach((el) => {
      if (styled.has(el)) return;  // apply once, even across polls
      styled.add(el);
      const baseDelay = parseFloat(el.getAttribute('aa-delay')) || 0;
      if (baseDelay > 0) el.style.animationDelay = baseDelay + 's';
    });
  };

  // Poll instead of waiting for DOMContentLoaded: render-blocking scripts
  // (e.g. those Webflow injects) can delay that event well past these
  // timers, so gating on it would never let the fallback win. 
  const deadline = performance.now() + FALLBACK_DELAY + 8000;
  const tryFallback = () => {
    if (document.documentElement.hasAttribute('aa-loaded')) return;
    applyFallback();
    if (document.readyState === 'loading' && performance.now() < deadline) {
      setTimeout(tryFallback, 100);
    }
  };
  setTimeout(tryFallback, FALLBACK_DELAY);

  setTimeout(() => {
    if (document.documentElement.hasAttribute('aa-loaded')) return;
    document.documentElement.setAttribute('aa-timeout', '');
  }, TIMEOUT_DELAY);
</script>
<style>
  [aa-animate]:not([aa-ready]) { visibility: hidden; }
  @keyframes aa-fallback-appear {
    from { opacity: 0; transform: translateY(1.5rem); }
    to   { opacity: 1; transform: translateY(0); }
  }
  html[aa-fallback] [aa-trigger~="load"][aa-animate]:not([aa-ready]),
  html[aa-fallback] [aa-trigger~="load-once"][aa-animate]:not([aa-ready]) {
    visibility: visible;
    animation: aa-fallback-appear 0.5s ease both;
  }
  html[aa-timeout] [aa-animate]:not([aa-ready]) { visibility: visible; }
</style>
```

Tune the two constants at the top to taste. Keep `TIMEOUT_DELAY` well above `FALLBACK_DELAY` — the first is "let JS win the race for the hero", the second is "give up, show the page".

**Why the inline `<style>`** — the same FOUC + fallback rules ship in `dist/alrdy-animate.css`, but that file lands *after* the HTML renders — always on a slow connection, and deliberately so when you load it non-blocking (the [Webflow install guide](/installation/webflow/#1-site-wide-head-code) uses `media="print"` to keep it off the critical render path). Without inlining, every animated element flashes briefly visible, then disappears when the lib's CSS arrives, then fades back in when the timeout fires. Inlining the critical CSS in `<head>` closes that window — the FOUC guard is active from the very first paint, which is also what makes loading the external stylesheet non-blocking safe.

## How the staircase works

The script honours one timing attribute so the fallback looks like a quieter version of the real animation:

- **`aa-delay="<seconds>"`** — delays the element's fade-up. E.g. a paragraph with `aa-delay="0.2"` appears 200ms after the headline, so the hero's staircase still reads. CSS alone can't apply this (typed `attr()` isn't widely supported yet), so the script sets it inline.

Everything else fades as a whole element via the CSS keyframe. The script deliberately **ignores `aa-stagger`**: a stagger targets units that don't exist yet at fallback time — the split-text lines/chars of a `text-*` animation, or the children the real init walks — and the bundle hasn't run to create them. Trying to reproduce it risks animating the wrong nodes (e.g. only an inline `<span>` inside a headline, leaving the rest of the text to just appear). A clean whole-element fade is the safe quieter version; on a normal-speed load the bundle wins the race and the real per-line / per-child stagger plays instead.

## What each layer does

`init()` sets `aa-loaded` on `<html>` as its first synchronous step. The two timers check for that attribute and bail out if it's already set.

### Layer 1 — `aa-fallback` (1s default)

Sets `html[aa-fallback]`. The CSS rule matches both `[aa-trigger~="load"]` and `[aa-trigger~="load-once"]`, so it covers every load-triggered element (including combined forms like `aa-trigger="load-once event:enter"`). The keyframe fades those elements up, respecting `aa-delay` and `aa-stagger` so the staircase still plays.

When the bundle eventually arrives, the lib's load-trigger branches detect `aa-fallback` and skip their JS animation entirely — the elements stay where the fallback put them, no replay or flash. `init()` clears `aa-fallback` at the end, so subsequent Barba navigations don't re-trigger the keyframe.

Scoped to `[aa-trigger~="load"]` / `[aa-trigger~="load-once"]` on purpose: scroll-triggered elements below the fold stay hidden until the lib actually arrives. Only elements the author has explicitly marked as "needs to be visible immediately" are eligible for the fallback fade.

### Layer 2 — `aa-timeout` (4s default)

Sets `html[aa-timeout]`. The CSS rule reveals **every** `[aa-animate]` element at its natural rendered state — no animation, no from-state. This is the deep-fail safety net: if JS errored, the bundle is blocked, or the network is so slow that even the hero staircase isn't enough, the page still ends up usable.

When `init()` eventually arrives, it detects `aa-timeout` and switches to bypass mode:

- Drops `scroll`, `text`, `reveal`, `parallax` from the feature load set — no from-states applied, no entrance ScrollTriggers, no flash from rewinding the now-visible elements.
- Skips the reduced-motion fade pass.
- Still wires up interactive features (tabs, slider, marquee, modal, nav, hover, cursor, split) so components remain functional.
- Still mounts smooth-scroll, scroll-state, and scroll-target globals.
- Flips `aa-ready` on every `[aa-animate]` element so the DOM ends in a consistent state.
- Clears `aa-timeout`, so the next Barba navigation runs animations normally.

The tradeoff: appearance animations don't play on this load (the user already saw the content for 4+ seconds), and event-triggered scroll/text/reveal entries don't fire on this page. At that point, page consistency outweighs animation correctness.

## Customising the motion

The default keyframe is a small upward translate (`translateY(1.5rem) → 0`) plus opacity. To use a different fallback motion (slide-down, pure fade, scale-in), override the keyframe in your own CSS *after* the inline snippet:

```css
@keyframes aa-fallback-appear {
  from { opacity: 0; transform: scale(0.95); }
  to   { opacity: 1; transform: scale(1); }
}
```

The duration, easing, and selector stay the same — just the keyframe content changes.

## Why the snippet polls instead of waiting for `DOMContentLoaded`

The obvious way to write the apply step is "wait until the DOM is ready, then style the hero" — gate on `document.readyState` and fall back to `DOMContentLoaded`. **Don't.** On a real page it loses the race every time, for a reason that isn't obvious:

A render-blocking `<script src>` (no `defer`/`async`) pauses HTML parsing until it downloads. While the parser is paused, parsing is not "complete", so `document.readyState` stays `'loading'` and `DOMContentLoaded` does not fire. On a Webflow page there are always render-blocking scripts you don't control — Typekit in `<head>`, and the **jQuery + `webflow.js` + GSAP/ScrollTrigger** bundle Webflow appends at the *end* of `<body>`. On a genuinely slow connection those keep `readyState` at `'loading'` for **5–15 seconds**, and push `DOMContentLoaded` *past* `TIMEOUT_DELAY` — so the safety-net reveal has already fired and the staircase never plays. A `readyState`/`DOMContentLoaded` gate simply cannot win.

The fix is to stop using "document finished parsing" as a proxy for "the hero exists." The hero is near the top of `<body>` and is parsed within the first second or two regardless of what's blocking further down the page. So the snippet **polls for the load-trigger elements** and applies the moment they appear — typically seconds before `DOMContentLoaded` on a slow load.

### Still add `defer` to your own CDN scripts

`defer` doesn't fix the snippet (you can't defer Webflow's injected scripts), but it's still worth doing on the `<script src="…">` tags *you* control:

```html
<script defer src="https://cdn.jsdelivr.net/npm/gsap@3/dist/gsap.min.js"></script>
<script defer src="https://cdn.jsdelivr.net/npm/gsap@3/dist/ScrollTrigger.min.js"></script>
<script defer src="https://cdn.jsdelivr.net/npm/lenis@1/dist/lenis.min.js"></script>
<script defer src="https://cdn.jsdelivr.net/npm/alrdy-animate@{{version}}/dist/alrdy-animate.umd.js"></script>
```

Deferred scripts don't block parsing, so the hero parses sooner (the staircase can start earlier) and `init()` runs sooner once the bundle lands. Two related wins on Webflow specifically: load alrdy-animate's GSAP build *instead of* Webflow's native GSAP rather than both — shipping GSAP twice is a large chunk of the render-blocking weight that delays the real init — and if you can, bind your `AlrdyAnimate.init(...)` call to the bundle being ready rather than to `DOMContentLoaded`, which on a slow Webflow load is one of the last events to fire.

## When you don't need this

You can skip the snippet if any of these are true:

- The bundle ships in your own JS bundle (Vite / Next.js / Webpack), not from a CDN. In that case bundle availability is bound to page render, not network latency, and a JS error would prevent the React tree itself from mounting — there's no `aa-animate` flash to protect against.
- You're shipping behind aggressive caching where most visitors hit a warm bundle *and* you're comfortable with elements staying invisible on the rare error case.

If you only want one of the two layers, delete the other layer's `setTimeout(...)` (for `aa-fallback`, the `tryFallback` block and its scheduling `setTimeout`) and its corresponding CSS rule.

## Page transitions (Barba)

This snippet only protects the *first* page load. On subsequent navigations the bundle is cached and `init()` runs in `beforeEnter` parallel to the leave timeline (see the [Webflow + Barba recipe](/recipes/webflow-barba/)). `aa-loaded` persists across navigations, so the fallback timer's check passes immediately and nothing fires.

## Verifying it locally

### Layer 1 — `aa-fallback` (slow network)

1. Open DevTools → Network → check "Disable cache" → throttle to "Slow 3G" (or "Slow 4G").
2. Reload. You should see:
   - Hero elements stay invisible for `FALLBACK_DELAY` plus however long the hero takes to parse — roughly 1–3s on a slow connection, not a hard 1s.
   - They then fade up — respecting any `aa-delay` and `aa-stagger` you set, so the staircase plays out before the bundle finishes downloading.
   - This should land *before* the `aa-timeout` reveal (4s default). If the staircase only ever appears as an instant, no-animation reveal at the 4s mark, the fallback isn't winning — see the troubleshooting note below.
   - When the bundle eventually lands, no second animation plays — the elements stay where the fallback put them.
3. Open the Elements panel and inspect a load-trigger element while the fallback is in flight. You should see `<html aa-fallback>`, and on any element with `aa-delay`, an inline `style="animation-delay: 0.2s"`. Elements without `aa-delay` fade via the CSS keyframe with no inline style. If `aa-fallback` is set but a delayed element's inline style is missing, the timer fired before that element parsed — the poll retries and applies it once it appears.

If the staircase never plays and you only get the blanket 4s reveal, you almost certainly have an older copy of this snippet that gated the apply step on `document.readyState === 'loading'` / `DOMContentLoaded`. On any page with render-blocking scripts (every Webflow page) that gate defers the apply until `DOMContentLoaded`, which fires *after* the timeout — re-copy the current snippet, which polls instead.

### Layer 2 — `aa-timeout` (bundle never arrives)

1. In DevTools → Network, **block** the alrdy-animate `<script>` URL (right-click the request → Block request URL).
2. Reload. You should see:
   - All `[aa-animate]` elements stay hidden for ~4s (the `TIMEOUT_DELAY` interval).
   - At the timer mark, every animated element on the page becomes visible at its natural rendered state — no fades, no entrance motion.
   - The Elements panel shows `<html aa-timeout>`.
3. Unblock the request and reload. Hero fades in via Layer 1 if it gets stuck under 4s, otherwise normal init runs and the snippet does nothing.
